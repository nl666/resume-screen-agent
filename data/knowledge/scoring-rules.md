# 评分规则详解

## 为什么简历可能只有 55 分？

55 分属于 weak_match（50-64 分区间）。常见原因：

### 典型丢分点

1. **缺少企业级项目经验**：只有个人项目或课程项目，没有上线的企业级 AI 系统经验
2. **没有后端工程化经验**：未提及 Docker、K8s、CI/CD、Redis、Kafka 等基础设施
3. **Agent 框架缺失**：未使用 LangGraph、Dify、AutoGen 等框架，仅有简单的 API 调用
4. **RAG 经验不足**：未涉及 Chunk、Embedding、向量检索、Rerank 等
5. **Tool Calling/MCP 空白**：未提及 Function Calling、工具封装或系统集成
6. **工程化实践缺失**：无日志、监控、评测、成本优化等生产环境经验
7. **项目证据不充分**：缺少个人职责描述、技术栈细节、量化结果

### 封顶规则影响

- 如果没有真实 AI/Agent/RAG 项目证据，即使后端很强，总分最高 60-65 分
- 55 分通常意味着：有编程基础和 API 调用经验，但缺乏深度和广度

### 提升建议

- 补充具体的 Agent 项目，描述技术栈、架构、个人职责、上线效果
- 学习并使用至少一种 Agent 框架（LangGraph 推荐）
- 增加 RAG 实战经验（文档解析 → Chunk → Embedding → 检索 → 生成）
- 学习 Function Calling / Tool Calling
- 补充工程化知识（Docker、日志、监控、评测）

## 给分规则

- **0 分**：未提及，或明确写着未实现、尚未接入、没有相关能力
- **0%-20% 分值**：计划、后续、预留、可扩展、准备接入等规划型表述
- **30%-50% 分值**：学习、了解、熟悉、入门、自学等学习型表述
- **50%-60% 分值**：只提到关键词，但缺少明确动作、项目职责或结果
- **70%-90% 分值**：使用过、参与过、基于某技术完成项目，但细节不完整
- **100% 分值**：已实现 / 已构建 / 已封装 / 已上线，有项目 + 技术栈 + 个人职责 + 结果证据

## 证据强度分档

系统会先判断简历片段的证据强度，再按强度调整得分：

| 证据强度 | 典型表述 | 评分处理 |
|---|---|---|
| 已落地 implemented | 实现、完成、构建、搭建、封装、开发、上线、运行、校验 | 可按关键词覆盖情况给高分 |
| 使用过 used | 使用、基于、接入、集成、采用、负责、参与、设计 | 给中高分，但低于明确已落地 |
| 仅提及 mentioned | 只出现关键词，没有动作证据 | 给较低分 |
| 学习了解 learning | 学习、了解、熟悉、入门、自学、有兴趣 | 给低分，不算充分项目证据 |
| 计划中 planned | 计划、后续、预留、可扩展、准备接入、未来 | 给极低分，不算已落地证据 |
| 未实现 not_implemented | 未实现、尚未接入、没有、缺少、不足 | 0 分，不算能力覆盖 |

风险和缺失项只认可“已落地”和“使用过”两类证据。计划中、学习中、仅提及、未实现都不会被当成真实能力覆盖。

## 硬性门槛

4 项必须满足至少 3 项才算 pass：
1. 有 Python / Java / Go / TypeScript 中至少一门主力开发语言
2. 有后端开发、AI 应用开发或系统集成经验
3. 有 LLM / Agent / RAG / Tool Calling / Prompt 工程相关项目经验
4. 能看到真实项目证据（上线项目、企业项目、Side Project、GitHub、技术文章）

- 满足 3+ 项 → pass
- 满足 2 项 → unclear
- 满足 0-1 项 → fail（直接 not_recommended）
